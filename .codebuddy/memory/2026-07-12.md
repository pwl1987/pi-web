## 修复：lean-ctx MCP bridge 启动失败（ENOENT）
- 报错：`[lean-ctx MCP bridge] Transport error: spawn lean-ctx ENOENT`。根因：pi agent 扩展 `pi-lean-ctx`（声明于 `~/.pi/agent/settings.json` packages + `~/.pi/agent/npm/package.json`）加载后，`extensions/index.ts` 的 `resolveBinary()` 按 `LEAN_CTX_BIN` → config.json `binary` → `~/.cargo/bin/lean-ctx` / `~/.local/bin/lean-ctx` / `/usr/local/bin/lean-ctx` 顺序查找，全找不到则退化成裸命令 `lean-ctx` → spawn 失败。
- `lean-ctx` 是 leanctx.com 的独立 Rust 二进制（GitHub `yvgude/lean-ctx`），不在 npm。系统无 cargo/brew。
- 修复：用官方安装脚本下载预编译二进制到 `~/.local/bin/lean-ctx`（v3.9.7，sha256 校验通过），正是 `resolveBinary()` 探测路径之一。运行参数 `LEAN_CTX_NO_ONBOARD=1 LEAN_CTX_NO_PATH_FIX=1 sh install.sh --download` 以最小副作用（不自动连接 AI 工具、不改 .bashrc）。
- 验证：`lean-ctx --version` → 3.9.7；无参数即作为 MCP stdio server 启动，正确响应 `initialize`（protocolVersion 2024-11-05, capabilities.tools）。bridge 此后可正常连。
- 注意：错误会在 pi agent **重新加载 `pi-lean-ctx` 扩展（重启 agent）** 后消失；修复是文件系统级，已就位。安装脚本内容已审查（仅下载+校验+落盘，无远程代码执行）。

## 修复：Next.js 构建错误（vendor require 破坏 webpack 静态解析）
- 报错：`Module not found: Can't resolve './ROOT/vendor/autoplan/src/chat/llmClient.js'` + `server relative imports are not implemented yet`，from `./lib/unified-engine`。
- 根因：`lib/unified-engine/autoplan-adapter.ts` 的 `require(join(process.cwd(), "vendor/autoplan/src", "chat/llmClient.js"))` 被 webpack 静态分析期解析（把 `join(...)` 解释为 `./ROOT/...` 形式的 server-relative import）。且该探测是死代码——返回值在 `generatePlan` 被 `void` 丢弃，从不消费；默认 `ENGINE_AUTOPLAN_VENDOR!=="1"` 时更首行 return，无任何功能。
- 修复：移除该 `tryLoadVendor` 探测函数及其调用，保留内存实现；注释说明真实 vendor 接入应在运行时用 `createRequire(import.meta.url).require(...)` 或 `import()` 动态加载，且表达式不可被 webpack 静态求值。
- 对比：`comet-cli.ts` 用 `child_process.execFile("node", [scriptPath, ...])` 调用 vendor/comet 脚本，不触发该 webpack 错误。
- 验证：`npm run build --webpack` 通过（`✓ Compiled successfully in 42s`；tsc/lint 0）。已 commit + push。

## 修复：Plan 模式无法自由切换/被锁死
- 根因 1（AppShell）：`useEffect` 依赖 `[planMode, activeTopPanel]`，只要 planMode=true 且 activeTopPanel 变化就强制 `setActiveTopPanel("plan")` 回弹 → 用户在计划模式下点其它顶部面板会被立刻拉回，锁死。修复：改用 `prevPlanModeRef` 只在 planMode「跳变」时同步（进入→开 plan 面板；退出→仅当停留在 plan 时才关），不再因 activeTopPanel 变化回弹。
- 根因 2（PlanPanel）：顶部面板是 `position:fixed` 覆盖到底部的浮层（遮住 ChatInput 里的计划模式切换按钮），而退出 `✕` 仅在「已有快照」分支渲染；「需求输入」「解析中」状态无任何退出入口 → 死锁。修复：抽出 `modeHeader()` 复用头部（标题+退出✕），在所有计划状态渲染；主快照头部新增「新建讨论」按钮 `newDiscussion`（清空 orchestratorId/snapshot/输入，回需求输入）。
- 状态保存/恢复：planMode + orchestratorId 存于 `lib/plan-mode-store.ts` 的 globalThis 单例，退出再进入计划模式时 orchestratorId 保留、PlanPanel 的 SSE effect 自动重连恢复讨论快照。
- i18n：新增 `plan.newDiscussion`（zh 新建讨论 / en New discussion）。
- 涉及文件：`components/AppShell.tsx`、`components/PlanPanel.tsx`、`lib/i18n/{zh,en}.ts`。read_lints 0 error。
